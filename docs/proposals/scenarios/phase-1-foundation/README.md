# Phase 1: Foundation and Example Maintainability

## Purpose

Phase 1 focuses on making Ianvs scenario work easier to validate, understand, and maintain. The goal is to strengthen example reliability before broader benchmark expansion.

## Scope

This phase groups proposals that center on restoring representative examples, clarifying maintenance expectations, and reducing the risk that scenario work drifts away from executable Ianvs behavior.

## Included Proposals

| Proposal | Role in This Phase | Reason |
| --- | --- | --- |
| `example-restoration` | Core foundation proposal | It explicitly focuses on restoring examples, improving maintainability, and establishing a more reliable baseline for future scenario work. |
| `Example_Restoration` | Related restoration proposal | It covers a closely related restoration direction and should be considered together with `example-restoration` when defining restoration priorities and contributor guidance. |

## Expected Deliverables

- Restored representative examples that can serve as maintainable references
- Clear restoration scope and prioritization guidance
- Example health classification or maintenance criteria for proposal review
- Maintainer-facing guidance for handling broken, outdated, or risky examples
- Contributor-facing guidance for restoration-oriented scenario work

## Relationship to Other Phases

Phase 1 provides the minimum reliability layer for later benchmark planning. Without a clearer restoration and maintenance baseline, reusable benchmarks in Phase 2 and advanced scenarios in Phase 3 are harder to validate consistently.

## Notes and Constraints

This phase is not described as a full solution for every Ianvs example. Its role is to improve confidence in representative examples and to define a maintainable path for future scenario growth.

The overlap between `example-restoration` and `Example_Restoration` is inferred from the folder names and proposal direction. They should be reviewed together for potential consolidation or role separation.
